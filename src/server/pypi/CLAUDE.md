# PyPI Protocol Implementation

## Overview
PyPI (Python Package Index) server implementing PEP 503 Simple Repository API. The LLM controls package availability, version lists, and file serving while hyper handles the underlying HTTP protocol.

**Status**: Experimental
**RFC**: PEP 503 (Simple Repository API)

## Library Choices
- **hyper v1.0** - HTTP server library (same as HTTP protocol)
- **hyper-util** - Additional utilities for Tokio integration
- **http-body-util** - Body handling utilities
- **Manual PEP 503 compliance** - LLM generates HTML responses according to PEP 503 format

**Rationale**: PyPI is fundamentally an HTTP server with specific path conventions and HTML response formats. We reuse the hyper infrastructure from the HTTP protocol implementation but add PyPI-specific logic:
- Path parsing to identify request types (list packages, list files, download)
- Request classification for the LLM
- PEP 503 HTML format guidance in event descriptions

The LLM focuses on application logic (which packages exist, which versions are available, what file contents to serve) rather than HTTP or HTML formatting details.

## PEP 503 Simple Repository API

### Required Endpoints

1. **Root/Index endpoint** (`/simple/`):
   - Returns HTML5 page listing all available packages
   - Each package as an anchor tag: `<a href="package-name/">package-name</a>`

2. **Project endpoint** (`/simple/<package-name>/`):
   - Returns HTML5 page listing all distribution files for the package
   - Each file as an anchor tag with href and optional hash: `<a href="../../packages/path/file.whl#sha256=...">file.whl</a>`

3. **File download endpoint** (`/packages/<path>/<filename>`):
   - Returns the actual package file (wheel, sdist, etc.)
   - Content-Type should be appropriate for the file type

### URL Requirements (PEP 503)
- All HTML endpoints MUST end with `/`
- Project names should be normalized (lowercase, replace `_` and `-` with `-`)
- File URLs can be absolute or relative

### HTML Format
Both index and project pages must be valid HTML5 with anchor elements. Minimal example:
```html
<!DOCTYPE html>
<html>
<body>
<a href="package-name/">package-name</a>
</body>
</html>
```

## Architecture Decisions

### 1. HTTP-Based Implementation
PyPI is built on top of HTTP/1.1:
- Uses hyper for HTTP protocol handling (same as HTTP protocol)
- Each HTTP request is classified as `list_packages`, `list_files`, or `download_file`
- LLM generates appropriate responses based on request type

### 2. Request Classification
The server analyzes the request path to determine the type:
- `/` or `/simple/` → `list_packages`
- `/simple/<package>/` → `list_files` (package name extracted)
- `/packages/<path>/<filename>` → `download_file` (filename extracted)
- Other paths → `unknown`

This classification is provided to the LLM in the `request_type` parameter.

### 3. Connection Tracking
Similar to HTTP protocol:
- One `ConnectionId` per TCP connection (not per HTTP request)
- `ProtocolConnectionInfo::Pypi` tracks recent requests on the connection
- Multiple PyPI requests can occur on the same TCP connection (HTTP keep-alive)

### 4. Body Handling
Same as HTTP protocol:
- Request bodies fully buffered (though typically empty for GET requests)
- Response bodies generated by LLM as complete strings
- No streaming support

### 5. Dual Logging
All PyPI operations use dual logging:
- **DEBUG**: Request summary (method, URI, request type)
- **TRACE**: Full request details (headers)
- Both go to `netget.log` (via tracing) and TUI Status panel (via status_tx)

### 6. Error Handling
LLM errors result in 500 Internal Server Error:
- If LLM call fails, return 500 with "Internal Server Error" body
- If LLM response is invalid, return default 200 OK with empty body
- Hyper connection errors close connection automatically

## LLM Integration

### Action-Based Response Model
The LLM responds to PyPI events with actions:

**Events**:
- `pypi_request` - PyPI HTTP request received from client (pip, twine, etc.)
  - Parameters: `method`, `uri`, `path`, `headers`, `body`, `request_type`, `package_name`

**Available Actions**:
- `send_pypi_response` - Send PyPI HTTP response (status, headers, body)
- Common actions: `show_message`, `update_instruction`, etc.
- **No async actions** - PyPI is purely request-response

### Request Types

The LLM receives a `request_type` parameter indicating what kind of request this is:

1. **`list_packages`** - Client requests `/simple/`
   - LLM should return HTML listing all available packages
   - Each package as `<a href="package-name/">package-name</a>`

2. **`list_files`** - Client requests `/simple/<package-name>/`
   - `package_name` parameter contains the package name
   - LLM should return HTML listing all available files for that package
   - Each file as `<a href="../../packages/hash/filename#sha256=...">filename</a>`

3. **`download_file`** - Client requests `/packages/<path>/<filename>`
   - `package_name` parameter contains the filename
   - LLM should return the actual file content
   - Can be binary data (wheels, tarballs) or text (setup.py, PKG-INFO)

### Example LLM Response (list_packages)
```json
{
  "actions": [
    {
      "type": "send_pypi_response",
      "status": 200,
      "headers": {
        "Content-Type": "text/html"
      },
      "body": "<!DOCTYPE html>\n<html>\n<body>\n<a href=\"hello-world/\">hello-world</a>\n<a href=\"example-pkg/\">example-pkg</a>\n</body>\n</html>"
    },
    {
      "type": "show_message",
      "message": "Served package index"
    }
  ]
}
```

### Example LLM Response (list_files)
```json
{
  "actions": [
    {
      "type": "send_pypi_response",
      "status": 200,
      "headers": {
        "Content-Type": "text/html"
      },
      "body": "<!DOCTYPE html>\n<html>\n<body>\n<a href=\"../../packages/abc/hello_world-1.0.0-py3-none-any.whl#sha256=abc123\">hello_world-1.0.0-py3-none-any.whl</a>\n</body>\n</html>"
    }
  ]
}
```

### Example LLM Response (download_file)
```json
{
  "actions": [
    {
      "type": "send_pypi_response",
      "status": 200,
      "headers": {
        "Content-Type": "application/zip"
      },
      "body": "PK\x03\x04..."
    }
  ]
}
```

### Response Format
The `send_pypi_response` action returns structured data:
```json
{
  "status": 200,
  "headers": {"Content-Type": "text/html"},
  "body": "<!DOCTYPE html>..."
}
```

### Default Response
If LLM doesn't provide a response or response parsing fails:
- Status: 200 OK
- Headers: empty
- Body: empty string

This ensures the server always responds (no hanging connections), though pip may fail if it receives invalid responses.

## Connection Management

### Connection Lifecycle
1. **Accept**: `TcpListener::accept()` creates new TCP connection
2. **Register**: Connection added to `ServerInstance` with `ProtocolConnectionInfo::Pypi`
3. **Serve**: Hyper's `http1::Builder::new().serve_connection()` handles HTTP/1.1 protocol
4. **Request Loop**: Each request on the connection calls the service function
5. **LLM Call**: Service function calls LLM to generate response
6. **Close**: Connection removed when TCP socket closes (client disconnect or error)

### Connection Data Structure
```rust
ProtocolConnectionInfo::Pypi {
    recent_requests: Vec<String>, // Track recent URIs requested
}
```

### State Updates
- Connection state tracked in `ServerInstance.connections`
- Each request increments `packets_received` and `bytes_received`
- Each response increments `packets_sent` and `bytes_sent`
- `last_activity` updated on each request
- UI updates via `__UPDATE_UI__` message

## Known Limitations

### 1. No Binary File Support
- LLM must return package files as strings
- Binary files (wheels, tarballs) are challenging for LLMs to generate
- Workaround: LLM can return placeholder/dummy files or base64-encoded content

### 2. No Package Validation
- No verification that returned HTML conforms to PEP 503
- No validation of file hashes (sha256)
- No checking of package metadata format

### 3. No Authentication
- No support for authenticated package repositories
- No API tokens or user credentials
- Public repository only

### 4. No Package Upload
- PEP 503 only covers package download (Simple Repository API)
- Package upload (PEP 691, Warehouse API) not implemented
- Read-only repository

### 5. HTTP/1.1 Only
- No HTTP/2 support
- No TLS/HTTPS support (plain HTTP only)
- Should use localhost or private networks

### 6. No Caching
- No ETag or Last-Modified support
- No conditional requests
- Every request results in LLM call (slow)

### 7. Memory Constraints
- Package files fully buffered in memory
- Large packages (>100MB wheels) may cause issues
- No streaming file support

## Example Prompts

### Simple Single Package
```
listen on port 8080 via pypi
Serve a package called "hello-world" version 1.0.0
When pip requests package list, return hello-world
When pip requests hello-world files, return a wheel file: hello_world-1.0.0-py3-none-any.whl
When pip downloads the wheel, return a minimal zip file with PKG-INFO
```

### Multiple Packages
```
listen on port 8080 via pypi
Serve three packages:
1. "requests" with version 2.28.0
2. "numpy" with versions 1.24.0 and 1.25.0
3. "pandas" with version 2.0.0
Return appropriate HTML for package list and file lists
Return placeholder wheel files when downloaded
```

### Conditional Package Availability
```
listen on port 8080 via pypi
Serve packages based on request headers:
- If User-Agent contains "pip", serve "example-pkg" version 1.0.0
- If User-Agent contains "poetry", serve version 1.1.0
- Otherwise return 404
```

### Dynamic Package Generation
```
listen on port 8080 via pypi
Act as a dynamic PyPI mirror:
- When pip requests any package, claim it exists
- Generate version 0.0.1 for any package
- Return minimal wheel files with just PKG-INFO metadata
```

## Performance Characteristics

### Latency
- One LLM call per HTTP request
- Typical latency: 2-5 seconds per request with qwen3-coder:30b
- pip makes multiple requests: index, package page, file download

### Throughput
- Limited by LLM response time (2-5s per request)
- pip install may take 10-15 seconds for single package (multiple requests)

### Concurrency
- Multiple pip clients can install concurrently
- Each request processed on separate tokio task
- Ollama lock serializes LLM API calls

### Memory Usage
- Each request/response buffered in memory
- Package files (wheels) can be large (10-100MB)
- Multiple concurrent requests may exhaust memory

## Use Cases

### 1. Internal Package Repository
Host internal/private Python packages:
```
listen on port 8080 via pypi
Serve our company's internal packages:
- "internal-auth" version 2.1.0
- "data-pipeline" version 1.5.3
- "ml-utils" version 3.2.0
```

### 2. PyPI Honeypot
Monitor pip install attempts:
```
listen on port 8080 via pypi
Act as a honeypot PyPI server
Log all package requests with IP addresses
Return fake packages to simulate a real repository
```

### 3. Testing & Development
Test pip behavior with unusual responses:
```
listen on port 8080 via pypi
Return random HTTP status codes for testing
Sometimes return valid packages, sometimes 404
Simulate slow responses (10+ seconds)
```

### 4. Educational/Demo
Demonstrate how PyPI works:
```
listen on port 8080 via pypi
Explain each pip request in plain English
Show what pip is asking for and why
Return minimal but valid responses
```

## Comparison with Raw HTTP

| Feature | HTTP (Raw) | PyPI |
|---------|-----------|------|
| Protocol | Generic HTTP | PEP 503 Simple Repository |
| Use Case | Web APIs, webhooks | Python package hosting |
| Request Format | Any | `/simple/` paths |
| Response Format | Any | HTML with anchor tags |
| Client | curl, browsers, etc. | pip, poetry, twine |
| Validation | None | PEP 503 compliance expected |

## Future Enhancements

### PEP 691 JSON API
Add JSON endpoint support:
- `/simple/` with `Accept: application/vnd.pypi.simple.v1+json`
- Return JSON instead of HTML
- More machine-readable than HTML

### Package Upload (Warehouse API)
Support package publishing:
- POST `/legacy/` endpoint (twine upload)
- Authentication with API tokens
- Package storage backend

### HTTPS Support
Add TLS for secure package downloads:
- Prevents MITM attacks
- Required for production PyPI mirrors

### Package Caching
Cache package metadata and files:
- Reduce LLM calls for repeated requests
- ETag and Last-Modified support
- Faster responses for pip

### Streaming Large Files
Support streaming package downloads:
- Don't buffer entire wheel in memory
- Allow >100MB packages
- Chunked transfer encoding

### Metadata Validation
Validate LLM-generated responses:
- Check PEP 503 HTML format
- Verify file hashes match
- Ensure proper URL structure
