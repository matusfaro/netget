# Maven Repository Protocol Implementation

## Overview

Maven repository server implementing the Maven repository HTTP API. The LLM controls artifact availability, content
generation (POM files, JAR files, checksums), and version metadata, while hyper handles the HTTP protocol and the server
parses Maven-specific URL paths.

**Status**: Experimental
**Base Protocol**: HTTP/1.1 (RFC 7230-7235)
**Specification**: [Maven Repository Layout](https://maven.apache.org/repository/layout.html)

## Library Choices

- **hyper v1.0** - Modern async HTTP library (same as HTTP protocol)
- **hyper-util** - Additional utilities for Tokio integration
- **http-body-util** - Body handling utilities
- **Manual Maven path parsing** - Custom parser for Maven artifact coordinates

**Rationale**: Maven repositories are fundamentally HTTP servers with specific path conventions. By building on top of
hyper (like the HTTP protocol), we get robust HTTP/1.1 support while adding Maven-specific path parsing and response
generation.

## Maven Repository Fundamentals

### Artifact Coordinates

Maven artifacts are identified by coordinates (GAV):

- **GroupId**: Reverse domain name (e.g., `com.example`)
- **ArtifactId**: Project identifier (e.g., `mylib`)
- **Version**: Version number (e.g., `1.0.0`)
- **Classifier** (optional): Build variant (e.g., `sources`, `javadoc`)
- **Extension**: File type (e.g., `jar`, `pom`, `war`)

### URL Path Structure

Maven clients request artifacts using this path pattern:

```
/{groupId}/{artifactId}/{version}/{artifactId}-{version}[-{classifier}].{extension}
```

GroupId dots are converted to slashes:

- `com.example` → `com/example`

Examples:

- `/com/example/mylib/1.0.0/mylib-1.0.0.jar` - Main JAR file
- `/com/example/mylib/1.0.0/mylib-1.0.0.pom` - Project metadata
- `/com/example/mylib/1.0.0/mylib-1.0.0-sources.jar` - Sources JAR
- `/com/example/mylib/1.0.0/mylib-1.0.0.jar.sha1` - SHA-1 checksum
- `/com/example/mylib/maven-metadata.xml` - Version listing

### File Types

**Core Files:**

- `.pom` - Project Object Model (XML metadata describing the artifact)
- `.jar` - Java Archive (compiled bytecode and resources)
- `.war` - Web Application Archive
- `.aar` - Android Archive

**Metadata:**

- `maven-metadata.xml` - Lists all available versions for an artifact

**Checksums** (for integrity verification):

- `.sha1` - SHA-1 hash
- `.md5` - MD5 hash
- `.sha256` - SHA-256 hash
- `.sha512` - SHA-512 hash

Each artifact file (`.jar`, `.pom`, etc.) typically has corresponding checksum files.

## Architecture Decisions

### 1. Request-Response Model

Like standard HTTP, Maven is synchronous request-response:

- Each artifact request is handled independently
- LLM decides if artifact exists and generates content
- No persistent state between requests
- No async actions - purely reactive to client requests

### 2. Path Parsing

Custom parser extracts Maven coordinates from URL paths:

- Validates path format follows Maven conventions
- Extracts `group_id`, `artifact_id`, `version`, `classifier`, `extension`
- Detects metadata requests (`maven-metadata.xml`)
- Detects checksum requests (`.sha1`, `.md5`, etc.)
- Invalid paths return 404 immediately without LLM call

### 3. Connection Tracking

Like HTTP, connections are tracked per TCP connection:

- One `ConnectionId` per TCP connection
- `ProtocolConnectionInfo::Maven` tracks recent artifact requests
- Multiple Maven requests on same TCP connection (HTTP keep-alive)
- Connection closed when TCP socket closes

### 4. Response Generation

The LLM can respond with three action types:

**send_maven_artifact**: Send artifact file content

- For `.jar`, `.pom`, `.war`, and other files
- Supports text and binary content
- Content-Type automatically set based on file type

**send_maven_metadata**: Send version listing XML

- Automatically generates `maven-metadata.xml` format
- LLM provides list of versions, latest version, release version
- XML structure generated by server

**send_maven_error**: Send error response

- Typically 404 Not Found for missing artifacts
- Can also send 403 Forbidden, 500 Internal Server Error, etc.

### 5. Dual Logging

All Maven operations use dual logging:

- **DEBUG**: Request summary (group ID, artifact ID, version, extension)
- **TRACE**: Full parsed artifact details
- Both go to `netget.log` (via tracing) and TUI Status panel (via status_tx)

### 6. Binary Content Handling

Maven repositories serve binary files (JARs):

- Response bodies can be text (for POM, XML, checksums) or binary (for JARs)
- Binary content can be base64-encoded in LLM response
- Server handles both string and base64 in action responses

## LLM Integration

### Action-Based Response Model

The LLM responds to Maven artifact requests with actions:

**Events**:

- `maven_artifact_request` - Maven client requests an artifact
    - Parameters: `method`, `uri`, `group_id`, `artifact_id`, `version`, `classifier`, `extension`, `is_metadata`,
      `is_checksum`, `checksum_type`, `headers`

**Available Actions**:

- `send_maven_artifact` - Send artifact file content
- `send_maven_metadata` - Send version listing XML
- `send_maven_error` - Send error response (404, etc.)
- Common actions: `show_message`, `update_instruction`, etc.
- **No async actions** - Maven is purely request-response

### Example LLM Responses

**Serving a JAR file:**

```json
{
  "actions": [
    {
      "type": "send_maven_artifact",
      "status": 200,
      "content_type": "application/java-archive",
      "body": "PK\u0003\u0004..." // JAR file content or base64
    },
    {
      "type": "show_message",
      "message": "Served JAR: com.example:mylib:1.0.0"
    }
  ]
}
```

**Serving a POM file:**

```json
{
  "actions": [
    {
      "type": "send_maven_artifact",
      "status": 200,
      "content_type": "application/xml",
      "body": "<?xml version=\"1.0\"?>\n<project>...</project>"
    }
  ]
}
```

**Serving version metadata:**

```json
{
  "actions": [
    {
      "type": "send_maven_metadata",
      "group_id": "com.example",
      "artifact_id": "mylib",
      "versions": ["1.0.0", "1.0.1", "1.1.0"],
      "latest": "1.1.0",
      "release": "1.1.0"
    }
  ]
}
```

**Serving a checksum:**

```json
{
  "actions": [
    {
      "type": "send_maven_artifact",
      "status": 200,
      "content_type": "text/plain",
      "body": "a1b2c3d4e5f6..."
    }
  ]
}
```

**Artifact not found:**

```json
{
  "actions": [
    {
      "type": "send_maven_error",
      "status": 404,
      "message": "Artifact not found"
    }
  ]
}
```

## Connection Management

### Connection Lifecycle

1. **Accept**: `TcpListener::accept()` creates new TCP connection
2. **Register**: Connection added to `ServerInstance` with `ProtocolConnectionInfo::Maven`
3. **Serve**: Hyper's `http1::Builder::new().serve_connection()` handles HTTP/1.1
4. **Request Loop**: Each Maven request on the connection calls the service function
5. **Parse**: Custom parser extracts Maven artifact coordinates from URL path
6. **LLM Call**: Service function calls LLM to determine response
7. **Close**: Connection removed when TCP socket closes

### Connection Data Structure

```rust
ProtocolConnectionInfo::Maven {
    recent_artifacts: Vec<String>, // Track recent artifact paths requested
}
```

### State Updates

- Connection state tracked in `ServerInstance.connections`
- Each request increments `packets_received` and `bytes_received`
- Each response increments `packets_sent` and `bytes_sent`
- `last_activity` updated on each request
- UI updates via `__UPDATE_UI__` message

## Known Limitations

### 1. HTTP/1.1 Only

- No HTTP/2 support
- No HTTPS/TLS (raw HTTP only)
- Connection upgrade not supported

### 2. No Streaming

- Artifact content fully buffered before sending
- Large JARs (>100MB) may exhaust memory
- No support for chunked responses

### 3. No Maven Deploy

- This is a read-only repository (GET requests only)
- No support for Maven deploy (PUT requests)
- No authentication or authorization

### 4. No SNAPSHOT Metadata

- Simplified `maven-metadata.xml` generation
- No timestamp-based SNAPSHOT versioning
- No `<snapshot>` section in metadata

### 5. No Repository Index

- No `.index` files or archetype catalog
- No search functionality
- Clients must know exact coordinates

### 6. Checksum Generation

- LLM must generate checksums manually
- No automatic checksum calculation
- Server doesn't validate checksums

## Example Prompts

### Simple Library Repository

```
listen on port 8080 via maven
Serve a library com.example:hello-world:1.0.0
The JAR contains a simple Hello class
The POM has minimal metadata
Return 404 for other artifacts
```

### Multi-Version Repository

```
listen on port 8081 via maven
Serve com.example:mylib with versions 1.0.0, 1.0.1, and 1.1.0
Each version has a JAR and POM
Return maven-metadata.xml listing all versions
Latest version is 1.1.0
```

### Repository with Sources and Javadocs

```
listen on port 8082 via maven
Serve com.example:toolkit:2.0.0
Include main JAR, sources JAR, javadoc JAR, and POM
Generate SHA-1 checksums for all files
```

### Private Repository with Access Control

```
listen on port 8083 via maven
Serve internal libraries: com.company:internal-api:3.0.0
Only serve artifacts if request includes X-Auth-Token header
Return 403 Forbidden for unauthorized requests
```

## Performance Characteristics

### Latency

- One LLM call per Maven artifact request (unless path is invalid)
- Typical latency: 2-5 seconds per request with qwen3-coder:30b
- Path validation happens before LLM call (instant 404 for invalid paths)

### Throughput

- Limited by LLM response time
- Concurrent requests processed in parallel
- Ollama lock serializes LLM API calls

### Memory Usage

- Each artifact buffered in memory
- Large JARs can be memory-intensive
- No streaming support

## Comparison with Standard HTTP

| Feature        | HTTP                   | Maven                                  |
|----------------|------------------------|----------------------------------------|
| Protocol       | Generic HTTP           | HTTP + Maven path conventions          |
| URL Parsing    | Pass-through to LLM    | Pre-parsed into Maven coordinates      |
| Response Types | Any content            | Artifacts, metadata, checksums, errors |
| Use Case       | Web APIs, generic HTTP | Java artifact distribution             |
| Complexity     | LLM implements routing | LLM implements repository logic        |

## Testing Strategy

### E2E Testing with mvn CLI

The primary E2E test uses the real Maven CLI (`mvn`) to:

1. Configure a custom repository pointing to NetGet
2. Declare a dependency in `pom.xml`
3. Run `mvn dependency:get` or `mvn compile`
4. Verify Maven successfully downloads the artifact from NetGet
5. Check downloaded JAR/POM content

This validates:

- Maven path parsing correctness
- HTTP compatibility with Maven client
- Artifact content integrity
- Metadata XML format correctness

### Test Efficiency Target

- **Target**: < 10 LLM calls per E2E test suite
- **Strategy**: Reuse single server instance for multiple artifact requests
- **Optimization**: Bundle related scenarios (JAR, POM, checksum) in one test

## Future Enhancements

### HTTPS Support

Add TLS support for secure artifact distribution:

- Use rustls to wrap TCP listener
- Support client certificate authentication
- Matches enterprise Maven repository requirements

### Maven Deploy Support

Support uploading artifacts (PUT requests):

- LLM decides whether to accept upload
- Store artifacts in configurable backend
- Generate checksums automatically

### SNAPSHOT Versioning

Full SNAPSHOT support:

- Timestamp-based SNAPSHOT versions
- `<snapshot>` section in maven-metadata.xml
- Latest SNAPSHOT resolution

### Automatic Checksum Generation

Calculate checksums automatically:

- SHA-1, MD5, SHA-256, SHA-512
- Serve checksum files without explicit LLM content
- Validate checksums on deploy

### Repository Mirroring

Proxy to real Maven Central:

- Cache artifacts locally
- LLM decides whether to mirror or block
- Useful for air-gapped or restricted environments
