# POP3 Server Implementation

## Overview

POP3 (Post Office Protocol version 3) server implementing basic RFC 1939 functionality for email retrieval. POP3 allows mail clients to download emails from a mailbox.

## Library Choices

- **Manual Implementation** - No external POP3 library used
- Raw TCP handling with tokio for async I/O
- Line-based protocol parsing using `AsyncBufReadExt`
- **TLS Support** - rustls and tokio-rustls for optional POP3S (implicit TLS)
- **Certificate Generation** - rcgen for self-signed certificates
- Chosen for maximum flexibility and LLM control over protocol behavior

## Architecture Decisions

### Connection Handling

- **Single Event Type**: `POP3_COMMAND_EVENT` handles all POP3 commands
- Commands are parsed line-by-line from the TCP stream
- Each command triggers an LLM call for action-based response
- Connection ID tracked for multi-connection support

### LLM Integration

- **Action-based responses** - LLM returns JSON actions for all protocol interactions
- **Greeting on connect** - Special `CONNECTION_ESTABLISHED` command triggers initial +OK greeting
- **State machine** - POP3 has three states (Authorization, Transaction, Update) managed implicitly by LLM
- **Protocol-aware actions** - Dedicated actions for POP3 responses (+OK, -ERR, STAT, LIST, RETR, etc.)

### Session Management

- No persistent session state beyond connection tracking
- POP3 authentication state (USER → PASS → authenticated) determined by LLM logic
- Email messages are generated by LLM, not stored in NetGet
- Each command is stateless from NetGet's perspective

### Response Actions

The LLM controls POP3 responses through these actions:

- `send_pop3_greeting` - +OK greeting banner
- `send_pop3_ok` - +OK responses
- `send_pop3_err` - -ERR error responses
- `send_pop3_stat` - +OK message_count total_size (mailbox status)
- `send_pop3_list` - +OK multiline list of message sizes
- `send_pop3_uidl` - +OK multiline list of unique IDs
- `send_pop3_retr` - +OK multiline email message content
- `send_pop3_top` - +OK multiline email headers and n body lines
- `send_pop3_message` - Custom POP3 response
- `wait_for_more` - Wait for next command
- `close_connection` - Terminate session

## POP3 Protocol

### States

1. **Authorization** - USER/PASS commands, authenticate user
2. **Transaction** - STAT/LIST/RETR/DELE commands, retrieve/delete messages
3. **Update** - QUIT command, apply deletions and close

### Common Commands

- `USER username` - Specify username
- `PASS password` - Provide password
- `STAT` - Get mailbox status (message count, total size)
- `LIST [msg]` - List message sizes
- `RETR msg` - Retrieve message content
- `DELE msg` - Mark message for deletion
- `TOP msg n` - Get message headers and n body lines
- `UIDL [msg]` - Get unique message IDs
- `NOOP` - No operation (keep-alive)
- `RSET` - Reset session (undelete messages)
- `QUIT` - Close connection

### Response Format

- **Success**: `+OK [message]`
- **Error**: `-ERR [message]`
- **Multiline**: `+OK\r\nline1\r\nline2\r\n.\r\n` (terminated with single dot)

## Connection Management

- Connections tracked in `AppState` (bytes sent/received, packet counts)
- Each connection spawns independent async task
- Write operations use `AsyncWriteExt` directly on split write half
- Read operations use `BufReader` for line-based parsing

## State Management

- **No protocol-specific state** - POP3 doesn't use `ProtocolConnectionInfo::Pop3`
- Connection lifecycle managed by tokio tasks
- Session state (authenticated, current mailbox) implicit in LLM conversation context

## TLS Support (POP3S)

- **Implicit TLS** - POP3S on port 995 (connection starts with TLS handshake)
- **Configurable** - Enable via `enable_tls: true` in open_server action options
- **Self-signed certificates** - Auto-generated using rcgen
- **Customizable certificates** - LLM can specify CN, SAN, validity, organization
- **Backward compatible** - TLS is optional, defaults to plain POP3

### Enabling POP3S

Use the `open_server` action with TLS options:

```json
{
  "type": "open_server",
  "protocol": "pop3",
  "port": 995,
  "options": {
    "enable_tls": true,
    "tls_common_name": "mail.example.com",
    "tls_san_dns_names": ["mail.example.com", "localhost"],
    "tls_validity_days": 365
  }
}
```

## Limitations

- **No STARTTLS support** - Only implicit TLS (POP3S) is supported, not STLS upgrade
- **No APOP** - Challenge-response authentication not implemented
- **No message persistence** - Messages generated by LLM, not stored
- **No PIPELINING** - Commands processed sequentially
- **No SASL** - Extended authentication not supported
- **No EXPIRE** - Message retention policy not implemented

## Examples

### Example LLM Prompt (Plain POP3)

```
listen on port 110 via pop3. Send greeting '+OK POP3 server ready'.
For USER command, respond with '+OK user accepted'.
For PASS command, respond with '+OK logged in'.
For STAT, respond with '+OK 3 1024' (3 messages, 1024 bytes total).
For LIST, send multiline response with message sizes.
For RETR 1, send email message with headers and body.
```

### Example LLM Prompt (POP3S with TLS)

```
listen on port 995 via pop3 with TLS enabled. Send greeting '+OK Secure POP3 ready'.
For USER alice, respond '+OK user accepted'.
For PASS secret, respond '+OK logged in, 2 messages'.
For LIST, send '+OK 2 messages\r\n1 512\r\n2 768\r\n.\r\n'.
For RETR 1, send the email message.
```

### Example LLM Response (Greeting)

```json
{
  "actions": [
    {
      "type": "send_pop3_greeting",
      "message": "POP3 server ready"
    }
  ]
}
```

### Example LLM Response (STAT)

```json
{
  "actions": [
    {
      "type": "send_pop3_stat",
      "message_count": 3,
      "total_size": 1024
    }
  ]
}
```

### Example LLM Response (LIST)

```json
{

    {

        {"id": 1, "size": 512},

    }
  ]
}
```
