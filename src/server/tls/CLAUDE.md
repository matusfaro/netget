# TLS Protocol Implementation

## Overview

Generic TLS (Transport Layer Security) server that provides encrypted transport for custom application protocols. The
LLM controls the application protocol while NetGet handles TLS encryption, certificate management, and connection
handling.

**Status**: Experimental (Core Protocol)
**RFC**: RFC 8446 (TLS 1.3), RFC 5246 (TLS 1.2)
**Port**: Configurable (common: 443, 8443, 9443)

## Library Choices

- **tokio-rustls** - Async TLS server implementation
    - Provides TLS acceptor for TCP streams
    - Supports TLS 1.2 and TLS 1.3
    - Integrates seamlessly with tokio async runtime
- **rustls** - Modern TLS library
    - Pure Rust implementation (memory-safe)
    - High performance and security-focused
    - Uses ring for cryptographic primitives
- **rcgen** - Certificate generation
    - Self-signed certificate creation
    - X.509 certificate and key pair generation
    - Configurable validity periods and subject names

**Rationale**: TLS is a fundamental security protocol that provides encryption, authentication, and integrity for
network communications. By separating the TLS transport layer from the application protocol, we enable the LLM to
implement any custom protocol on top of a secure channel (similar to how HTTPS is HTTP over TLS).

## Architecture Decisions

### 1. Generic Transport Layer

TLS provides only the encrypted transport:

- Accepts TCP connections
- Performs TLS handshake
- Exposes decrypted byte stream to LLM
- LLM implements application protocol on top
- No protocol-specific logic in TLS implementation

### 2. Connection Flow

Standard TLS connection lifecycle:

1. Accept TCP connection on specified port
2. Perform TLS handshake using self-signed certificate
3. Emit `tls_connection_opened` event to LLM
4. Read decrypted application data from TLS stream
5. Emit `tls_data_received` event to LLM with data
6. LLM responds with application protocol logic
7. Write encrypted data back to client
8. Loop for multiple exchanges or close connection

### 3. Self-Signed Certificates

TLS configuration:

- Certificates generated by `tls_cert_manager::generate_default_tls_config()`
- Self-signed for testing/development
- Default CN: "netget-dns-server", SAN: localhost, *.local
- 365-day validity period
- Clients must disable certificate verification for testing
- Production usage would require proper CA-signed certificates

### 4. Dual Logging

- **DEBUG**: TLS handshake events, data summaries, byte counts
- **TRACE**: Full hex dumps of decrypted data (request and response)
- Both go to netget.log and TUI Status panel
- Text data logged as strings, binary as hex

### 5. Connection State Machine

Per-connection state tracking:

- **Idle**: Ready to process data, no LLM call in progress
- **Processing**: LLM call active, queue incoming data
- **Accumulating**: LLM requested more data (WAIT_FOR_MORE action)

Prevents concurrent LLM calls on same connection, ensures data ordering.

### 6. Data Queueing

If data arrives during LLM processing:

- New data queued in connection state
- After LLM response, queued data processed
- Prevents data loss and maintains order
- Loop continues until queue empty

## LLM Integration

### Event Types

#### `tls_connection_opened`

Triggered when TLS handshake completes successfully.

Event parameters: None (just notification)

Available actions:

- `send_tls_data` - Send initial greeting/banner
- `close_this_connection` - Close connection

Use case: Send protocol-specific greeting (e.g., "HELLO\r\n", HTTP response, custom banner)

#### `tls_data_received`

Triggered when decrypted data is received from client.

Event parameters:

- `data` (string) - Received data (UTF-8 if printable, hex if binary)

Available actions:

- `send_tls_data` - Send response data
- `wait_for_more` - Accumulate incomplete message
- `close_this_connection` - Close connection

Use case: Parse application protocol, validate, respond

### Available Actions

#### `send_tls_data`

Send data over TLS connection (encrypted automatically).

Parameters:

- `data` (string, required) - Data to send

Example:

```json
{
  "type": "send_tls_data",
  "data": "HTTP/1.1 200 OK\r\nContent-Length: 13\r\n\r\nHello, World!"
}
```

#### `wait_for_more`

Wait for more data before responding (for incomplete messages).

Parameters: None

Example:

```json
{
  "type": "wait_for_more"
}
```

Use case: HTTP request without complete headers, fragmented protocol messages

#### `close_this_connection`

Close the TLS connection.

Parameters: None

Example:

```json
{
  "type": "close_this_connection"
}
```

### Example LLM Response

```json
{
  "actions": [
    {
      "type": "send_tls_data",
      "data": "HTTP/1.1 200 OK\r\nContent-Type: text/plain\r\nContent-Length: 7\r\n\r\nSuccess"
    },
    {
      "type": "show_message",
      "message": "Sent HTTP 200 OK response over TLS"
    }
  ]
}
```

## Connection Management

### Connection Lifecycle

1. **Accept**: TCP listener accepts connection
2. **TLS Handshake**: `TlsAcceptor::accept()` performs TLS negotiation
3. **Register**: Connection logged in INFO messages, added to ServerInstance
4. **Split Stream**: Read and write halves separated for concurrent I/O
5. **Event Loop**: Read data, call LLM, send responses
6. **Close**: Connection ends on client disconnect, error, or LLM close action

### State Management

- Each connection has independent state (Idle/Processing/Accumulating)
- Queued data per connection
- Memory field for LLM context (not currently used, future enhancement)
- Write half stored in Arc<Mutex<>> for safe concurrent access

### Concurrent Connections

- Multiple connections handled independently
- Each connection has separate tokio tasks for reading
- No connection limits or rate limiting (bounded by system resources)

## Known Limitations

### 1. Self-Signed Certificates Only

- No support for custom certificates (future enhancement)
- No CA-signed certificate loading
- No certificate rotation/renewal
- Testing requires disabling certificate verification on clients

**Workaround**: Modify `tls_cert_manager` to load custom certificates from files.

### 2. No Client Authentication

- Server doesn't request client certificates
- No mutual TLS (mTLS) support
- All clients accepted after TLS handshake

**Workaround**: Implement authentication at application protocol layer.

### 3. No Connection Pooling or Limits

- Each connection handled independently
- No connection limits or rate limiting
- Memory usage grows with concurrent connections
- No idle timeout (connections persist until client closes)

**Workaround**: Use operating system limits (ulimit, firewall rules).

### 4. No TLS Session Resumption

- Each connection requires full TLS handshake
- No session ticket support
- No session cache
- Higher latency for repeated connections

**Impact**: Minimal for long-lived connections, higher for short-lived.

### 5. Binary Data Handling

- Binary data encoded as hex strings for LLM
- LLM cannot efficiently parse binary protocols
- Best for text-based protocols (HTTP, SMTP, custom text protocols)

**Workaround**: Use structured text formats (JSON, XML) at application layer.

## Performance Characteristics

### Latency

- **TLS Handshake**: ~50-100ms (one-time per connection)
- **Per Request (with scripting)**: Sub-millisecond
- **Per Request (without scripting)**: 2-5 seconds (LLM call)
- **Encryption Overhead**: ~5-10 microseconds per message

### Throughput

- **With Scripting**: Thousands of requests per second per connection
- **Without Scripting**: ~0.2-0.5 QPS per connection (LLM-limited)
- **Multiple Connections**: Limited by CPU and memory

### Comparison to Plain TCP

- **Advantage**: Encrypted, secure, tamper-proof
- **Disadvantage**: Higher latency (TLS handshake, encryption overhead)
- **Use Case**: Any protocol requiring confidentiality (web, custom APIs, sensitive data)

### Scripting Compatibility

Excellent scripting candidate:

- Request/response pattern typical
- Deterministic responses possible
- High request volume common
- Application protocol can be pre-defined

## Security Considerations

### TLS Protection

- Prevents eavesdropping (confidentiality)
- Prevents tampering (integrity)
- Server authentication (with proper certificates)
- Forward secrecy (depends on cipher suite)

### Self-Signed Certificate Risks

- Default setup vulnerable to MITM attacks
- Clients must explicitly trust self-signed cert
- Not suitable for production without proper PKI
- Browser warnings for HTTPS usage

### Application Layer Security

- TLS only protects transport, not application logic
- LLM must implement authentication/authorization
- Input validation still required
- No protection against application-level attacks

### Recommended Practices

1. Use proper CA-signed certificates in production
2. Implement authentication at application layer
3. Validate all input data in LLM logic
4. Set reasonable timeout values
5. Implement rate limiting if needed

## Use Cases

### 1. Custom HTTPS API

Implement REST API with LLM-controlled endpoints:

- Parse HTTP requests (method, path, headers, body)
- Route to appropriate handler
- Return JSON responses
- Handle authentication tokens

### 2. Secure File Transfer Protocol

Custom file upload/download protocol:

- Command-based (UPLOAD filename size, DOWNLOAD filename)
- Binary data transfer over secure channel
- Progress tracking and resume support

### 3. Encrypted Chat Protocol

Real-time messaging over TLS:

- Text-based commands (SEND message, JOIN room)
- Broadcast to multiple clients
- Presence notifications

### 4. IoT Device Communication

Secure communication with embedded devices:

- JSON-based command/response
- Device telemetry collection
- Firmware update protocol

### 5. Custom Database Protocol

Implement simple database over TLS:

- SQL-like or NoSQL commands
- Secure data transmission
- Transaction support
